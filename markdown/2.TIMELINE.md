## PHASE 1: ROUTING & CONFIGURATION

### 1. Configuration Management

**Thứ Tự Implementation:**
```
1. Tạo configuration trong application.yml → "app"
2. Tạo Config classes để đọc properties từ YAML
3. Implement Repository để load và quản lý config
```

**Kiến Trúc File Structure:**

```
📁 Cấu trúc file để quản lý configuration như database:
    1️⃣ Entity Layer(/entity): Định nghĩa data model (object structure)
    2️⃣ Config Layer(/data/config): Load data từ application.yml
    3️⃣ Repository Interface (/interfaces)
    └─ Định nghĩa contract cho data access
    └─ Mục đích: Dễ dàng chuyển đổi từ file config → database
    4️⃣ Repository Implementation (/repository): Implement interface với concrete logic
    5️⃣ Service Layer (/service)
    └─ Business logic và data manipulation
    └─ Query, filter, transform data
```

**Benefits của kiến trúc này:**
- Dễ dàng chuyển đổi từ YAML config → Database
- Code không cần thay đổi khi đổi data source
- Tuân thủ SOLID principles (Dependency Inversion)

### 3. Path Matching (PathTrie)

**Mục đích:**
```
Sử dụng cấu trúc dữ liệu Trie để match request path với endpoint
đã đăng ký. Hỗ trợ path parameters (ví dụ: /users/{userId}).
```

**Cách hoạt động:**
```
Registered Endpoints:
  /users              → GET
  /users/{userId}     → GET, PUT, DELETE
  /users/{userId}/orders → GET

Trie Structure:
  root
   └─ users
       ├─ [GET] ← match "/users"
       └─ {userId}  (wildcard node)
           ├─ [GET, PUT, DELETE] ← match "/users/123"
           └─ orders
               └─ [GET] ← match "/users/123/orders"
```

**Match Results:**
```
Request: GET /users/123
  → MatchResult: FOUND (matched /users/{userId})

Request: POST /users/123
  → MatchResult: METHOD_NOT_ALLOWED (path exists, method không đúng)

Request: GET /products/123
  → MatchResult: NOT_FOUND (path không tồn tại)
```

### 4. Error Handling

**Kiến Trúc File Structure:**
```
📁 Error handling gồm 2 thành phần:
1️⃣ DTO Layer (/dto)
   └─ ErrorResponse.java         → Data model cho error JSON body
2️⃣ Utility Layer (/util)
   └─ ErrorHandler.java          → Shared component, dùng chung cho tất cả filters
```

**ErrorResponse DTO:**
```
    ErrorResponse (com.cathay.apigateway.dto)
    ├── status: int               → HTTP status code (400, 401, 403, 404...)
    ├── error: String             → HTTP reason phrase ("Forbidden", "Not Found"...)
    ├── message: String           → Chi tiết lỗi (5xx → "Internal Server Error")
    ├── path: String              → Request path gây lỗi
    └── timestamp: LocalDateTime  → Thời điểm xảy ra lỗi (auto = now())
```

**Example Error Response:**
```json
{
    "status": 403,
    "error": "Forbidden",
    "message": "Access denied: insufficient role level",
    "path": "/api/v1/identify-service/users",
    "timestamp": "2026-02-09T12:00:00"
}
```

### 5. CORS Configuration (Cross Origin Resource Sharing)

**CORS là gì?**

Browser mặc định áp dụng **Same-Origin Policy** — chỉ cho phép JavaScript gọi API
cùng origin (protocol + domain + port). Khi Frontend (React/Vue chạy trên `http://localhost:3000`) gọi API Gateway (`http://localhost:8080`), browser sẽ **block** response vì khác origin.

CORS là cơ chế cho phép server **khai báo** những origin nào được phép truy cập tài nguyên.

**🎯 Nhiệm Vụ:**
```
🔲 Khai báo allowed origins trong application.yml (linh hoạt giữa dev/staging/prod)
🔲 Tạo CorsPropertiesConfig (@ConfigurationProperties) bind từ yml
🔲 Tạo CorsConfig (@Configuration) register CorsWebFilter bean
🔲 Spring tự xử lý: preflight OPTIONS, validate origin, set CORS response headers
```

**📁 Cấu trúc files:**
```
application.yml                          ← Khai báo CORS properties
  └─ app.cors.allowed-origins, allowed-methods, ...

CorsPropertiesConfig.java               ← @ConfigurationProperties bind yml → Java object
  └─ data/config/CorsPropertiesConfig.java

CorsConfig.java                          ← @Configuration tạo CorsWebFilter bean
  └─ config/CorsConfig.java
```

**🔄 CORS Flow (Spring xử lý tự động):**

```
    ┌────────────────────────────────────────────────────────────────────┐
    │  Browser gửi request cross-origin                                  │
    │  Origin: http://localhost:3000                                     │
    └───────────────────────┬────────────────────────────────────────────┘
                            │
    ┌───────────────────────▼────────────────────────────────────────────┐
    │  1. Simple Request hay Preflight?                                  │
    │     • Simple: GET/POST với Content-Type đơn giản, no custom headers│
    │     • Preflight: PUT/DELETE/PATCH, hoặc có custom headers          │
    └──────┬───────────────────────────────────────┬─────────────────────┘
           │ Simple Request                        │ Preflight Required
           │                                       │
           │                        ┌──────────────▼────────────────────┐
           │                        │  2. Browser tự gửi OPTIONS        │
           │                        │     Access-Control-Request-Method │
           │                        │     Access-Control-Request-Headers│
           │                        └──────────────┬────────────────────┘
           │                                       │
    ┌──────▼───────────────────────────────────────▼────────────────────┐
    │  3. CorsWebFilter (Spring built-in) kiểm tra Origin               │
    │     • Origin có trong allowed-origins whitelist?                  │
    │     • Tự thêm Vary: Origin header                                │
    └──────┬───────────────────────────────────────┬────────────────────┘
           │ ✔ Allowed                             │ ✖ Not Allowed
           │                                       │
    ┌──────▼──────────────────────────┐    ┌───────▼───────────────────┐
    │  4a. Spring tự thêm headers:    │    │  4b. Không thêm headers   │
    │  Access-Control-Allow-Origin    │    │  Browser sẽ block response│
    │  Access-Control-Allow-Methods   │    │  → CORS error trên console│
    │  Access-Control-Allow-Headers   │    └───────────────────────────┘
    │  Access-Control-Max-Age         │
    │  Access-Control-Expose-Headers  │
    └──────┬──────────────────────────┘
           │
    ┌──────▼──────────────────────────┐
    │  5. Preflight? → Return 200 OK  │
    │     (Không qua Authen/Author)   │
    │     Simple?   → Filter Chain    │
    └─────────────────────────────────┘
```

**📋 CORS Response Headers (Spring tự set):**

| Header | Giá trị | Ý nghĩa |
|---|---|---|
| `Access-Control-Allow-Origin` | `http://localhost:3000` | Origin được phép (KHÔNG dùng `*` nếu có credentials) |
| `Access-Control-Allow-Methods` | `GET, POST, PUT, DELETE, PATCH, OPTIONS` | HTTP methods được phép |
| `Access-Control-Allow-Headers` | `Authorization, Content-Type, X-Request-ID, ...` | Request headers được phép |
| `Access-Control-Allow-Credentials` | `true` | Cho phép gửi cookies/JWT |
| `Access-Control-Max-Age` | `3600` | Cache preflight response 1 giờ (giảm OPTIONS requests) |

**⚠️ Lưu ý quan trọng:**
```
1. Access-Control-Allow-Origin KHÔNG được dùng wildcard (*) khi Allow-Credentials = true
   → Phải trả exact origin: "http://localhost:3000"
   → Spring CorsWebFilter xử lý đúng điều này tự động

2. Preflight (OPTIONS) request được Spring trả response NGAY
   → Không đi qua Authentication/Authorization (vì OPTIONS không mang JWT)
   → CorsWebFilter chạy trước filter chain, không cần lo order

3. Allowed origins cấu hình trong application.yml, không hardcode
   → Dễ thay đổi giữa dev/staging/production environments
   → Có thể dùng allowed-origin-patterns cho regex matching (*.example.com)

4. Access-Control-Max-Age giúp browser cache preflight response
   → Giảm số lượng OPTIONS requests, cải thiện performance
```

### Filter Chain Overview

```
Request Flow:
┌──────────────────────────────────────────────────────────────┐
│  Client Request                                              │
└────────────────────┬─────────────────────────────────────────┘

         ┌───────────▼───────────┐

         │ Validation Filter     │ Order: -20

         └───────────┬───────────┘

         ┌───────────▼───────────┐
```

### Filter 1: Validation Filter (Order: -20)

**🎯 Nhiệm Vụ:**
```
✅ Kiểm tra endpoint có tồn tại không (path + method) — via PathTrie (EndpointRegisterService)
✅ Validate method requirements (via MethodRuleService):
   • POST/PUT/PATCH: Require body (check exists + size), Content-Type
   • GET/DELETE: Validate query params (check exists + size)
✅ Kiểm tra required headers của từng endpoint (via EndpointHeaderRuleService)
✅ Validate header values: allowed patterns (regex), max length
✅ Phòng chống CRLF injection trong header values
✅ Block requests từ blocked domains
```

---

### Filter 2: Authentication Filter (Order: 0)

**🎯 Nhiệm Vụ:**
```
✅ Phân biệt endpoint: Public vs Protected (isPublic flag)
✅ Public endpoint:
   • Skip JWT validation
   • Add Internal API key vào headers
   • Forward to next filter
✅ Protected endpoint:
   • Check access token có tồn tại (Bearer header) → 401 nếu missing
   • Validate JWT token (signature, expiration)
   • Extract user info (userId, email, role)
   • Add user context vào request headers
   • Add Internal API key vào headers
   • Forward to next filter
```

**🔐 Authentication Flow:**
```
          ┌─────────────────────┐
          │   Incoming Request  │
          └──────────┬──────────┘
                     │
               ┌─────▼─────┐
               │ isPublic? │
               └──┬─────┬──┘
                  │     │
                 Yes    No
                  │     │
                  │  ┌──▼──────────────────┐
                  │  │  Has Access Token?  │
                  │  └──┬─────────────┬────┘
                  │    Yes            No
                  │     │             │
                  │     │    ┌────────▼────────┐
                  │     │    │ 401 Unauthorized│
                  │     │    │ (Missing Token) │
                  │     │    └─────────────────┘
                  │     │
                  │  ┌──▼──────────────────┐
                  │  │ Extract & Validate  │
                  │  └──┬─────────────┬────┘
                  │   Valid        Invalid
                  │     │             │
                  │     │    ┌────────▼────────┐
                  │     │    │ 401 Unauthorized│
                  │     │    │ (Invalid Token) │
                  │     │    └─────────────────┘
                  │     │
                  │  ┌──▼──────────────────┐
                  │  │    Add Headers      │
                  │  └──┬──────────────────┘
                  │     │
          ┌───────▼─────▼───────┐
          │     Add API Key     │
          └──────────┬──────────┘
                     │
          ┌──────────▼──────────┐
          │     Next Filter     │
          └─────────────────────┘
```

### Filter 3: Authorization Filter (Order: 1) ✅ DONE

**🎯 Nhiệm Vụ:**
```
- Resolve endpoint: path + method → EndpointsEntity (EndpointRegisterService)
- Skip authorization cho public endpoints (isPublic)
- Lấy user role từ header X-User-Role (set bởi Authentication Filter)
- Resolve role name → level qua RoleService.getRoleLevel()
- Level check: userLevel >= endpoint.minRoleLevel?
   ✔ Allow → Next Filter
   ✖ Deny → ErrorHandler → 403 FORBIDDEN
```

**🔑 Authorization Logic (Level-Based Hierarchy):**
```
                    ┌───────────────────────┐
                    │   Incoming Request    │
                    └───────────┬───────────┘
                                │
                                ▼
                ┌───────────────────────────────┐
                │  endpoint.isPublic() == true? │
                └───────┬───────────────┬───────┘
                        │               │
                     YES│               │ NO
                        │               │
                        ▼               ▼
                ┌──────────────┐ ┌──────────────────────────┐
                │ Next Filter  │ │ Read X-User-Role header  │
                │ (skip authz) │ │ (set by AuthN Filter)    │
                └──────────────┘ └────────────┬─────────────┘
                                              │
                                              ▼
                                ┌──────────────────────────┐
                                │ X-User-Role present?     │
                                └───────┬──────────┬───────┘
                                        │          │
                                     YES│          │ NO
                                        │          ▼
                                        │  ┌─────────────────────┐
                                        │  │ 403 FORBIDDEN       │
                                        │  │ "Missing User Role" │
                                        │  └─────────────────────┘
                                        ▼
                              ┌────────────────────────┐
                              │ Resolve User Level     │
                              │ RoleService            │
                              │  .getRoleLevel(role)   │
                              │ e.g. "ADMIN" → 4       │
                              └──────────┬─────────────┘
                                         │
                                         ▼
                          ┌────────────────────────────────┐
                          │ userLevel >= minRoleLevel ?    │
                          └───────┬────────────────┬───────┘
                               YES│                │ NO
                                  ▼                ▼
                          ┌──────────────┐ ┌────────────────────┐
                          │ Allow Access │ │ 403 FORBIDDEN      │
                          │ → Next Filter│ │ "Insufficient      │
                          └──────────────┘ │  role level"       │
                                           └────────────────────┘
```

**👥 Role Hierarchy (Level-Based):**
```
    Level 4: ADMIN    → Full access to all endpoints
    Level 3: MANAGER  → Management endpoints + lower
    Level 2: STAFF    → Staff-level endpoints + lower
    Level 1: USER     → User-specific endpoints only

    Rule: userRole.level >= endpoint.min_role_level → Allow
    Higher level = more permissions (inherits all lower levels)
```

### Filter 4: Rate Limiting Filter (Dual Implementation)

**🎯 Nhiệm Vụ:**
```
🔲 Kiểm tra số lượng token còn lại (remaining quota)
🔲 Cho phép request tiếp tục nếu còn token available
🔲 Từ chối request nếu vượt quá giới hạn → 429 Too Many Requests
🔲 Hỗ trợ rate limiting theo IP (public endpoints)
🔲 Hỗ trợ rate limiting theo Account ID (protected endpoints)
```

---
### 🌎 Filter 4a: IP-Based Rate Limit

**Order:** `-100` (chạy đầu tiên, trước tất cả filter khác)

**Vị trí:** Vị trí đầu tiên trong filter chain

**Mục tiêu:** All endpoints (public + protected)

**Mục đích:**
- Chống DDoS / bot flood ở tầng ngoài cùng
- Giảm tải traffic cho toàn bộ hệ thống phía sau

**Lưu ý thiết kế:**
> Không thắt chặt rate limit theo IP vì 1 địa chỉ IP có thể có rất nhiều user dùng chung
> (NAT, mạng công ty, wifi công cộng). Chỉ chặn khi traffic từ 1 IP **bất thường rõ ràng**.

**Cache Key:** `rate_limit:ip:{IP_ADDRESS}`

**Flow:**
```
    Request đến Gateway
           │
    ┌──────▼──────────────────────────────────────────────┐
 1. │ Extract Client IP                                    │
    │ • Ưu tiên: X-Forwarded-For header (proxy/LB)        │
    │ • Fallback: request.getRemoteAddress()               │
    └──────┬──────────────────────────────────────────────┘
           │
    ┌──────▼──────────────────────────────────────────────┐
 2. │ Check Token Bucket trên Redis                        │
    │ • Key: "rate_limit:ip:{IP_ADDRESS}"                  │
    │ • DECR tokens (atomic)                               │
    │ • tokens >= 0 → CÒN quota                            │
    │ • tokens <  0 → HẾT quota                            │
    └──────┬──────────────────────────────────────────────┘
           │
    ┌──────┴──────┐
    │             │
  tokens >= 0  tokens < 0
    │             │
 Next Filter   429 Too Many Requests
    │             + Header: X-RateLimit-Retry-After
```

**Cấu hình mặc định:**

| Tham số | Giá trị | Giải thích |
|---|---|---|
| Burst Capacity | 200 | Tối đa 200 request tích lũy |
| Replenish Rate | 50/s | Nạp 50 token mỗi giây |
| Token Cost | 1 | Mỗi request tốn 1 token |

---

### 👤 Filter 4b: Account-Based Rate Limit

**Order:** `10` (chạy **sau** Authentication/Authorization Filter → cần biết Account ID)

**Vị trí:** Sau Authentication/Authorization Filter

**Mục tiêu:** Protected endpoints (yêu cầu đã đăng nhập)

**Mục đích:**
- Giới hạn usage per user/account (quota management)
- Đảm bảo công bằng giữa các user, tránh 1 user chiếm hết tài nguyên

**Use Cases:**
- API usage quotas per subscription tier
- Prevent single user from overloading system
- Fair resource distribution among users

**Cache Key:** `rate_limit:account:{ACCOUNT_ID}`

**Flow:**
```
    Request đã qua Authentication Filter
           │
    ┌──────▼──────────────────────────────────────────────┐
 1. │ Extract Account ID từ Header                         │
    │ • Đọc X-Account-Id (set bởi Authentication Filter)   │
    │ ✖ Missing? → Skip filter (không áp dụng cho          │
    │   unauthenticated request)                           │
    └──────┬──────────────────────────────────────────────┘
           │
    ┌──────▼──────────────────────────────────────────────┐
 2. │ Check Token Bucket trên Redis                        │
    │ • Key: "rate_limit:account:{ACCOUNT_ID}"             │
    │ • DECR tokens (atomic)                               │
    │ • tokens >= 0 → CÒN quota                            │
    │ • tokens <  0 → HẾT quota                            │
    └──────┬──────────────────────────────────────────────┘
           │
    ┌──────┴──────┐
    │             │
  tokens >= 0  tokens < 0
    │             │
 Next Filter   429 Too Many Requests
    │             + Header: X-RateLimit-Remaining: 0
    │             + Header: X-RateLimit-Retry-After
```

**Cấu hình mặc định:**

| Tham số | Giá trị | Giải thích |
|---|---|---|
| Burst Capacity | 50 | Tối đa 50 request tích lũy per account |
| Replenish Rate | 10/s | Nạp 10 token mỗi giây |
| Token Cost | 1 | Mỗi request tốn 1 token |



### 🪣 Thuật Toán: Token Bucket với Redis

**Token Bucket là gì?**

Token Bucket là thuật toán rate limiting hoạt động theo nguyên lý "xô chứa token".
Mỗi request muốn đi qua phải "trả" 1 token. Nếu xô hết token → request bị từ chối.

**3 tham số cốt lõi:**

- **Burst Capacity** (dung lượng xô): Số token tối đa mà xô có thể chứa.

- **Replenish Rate** (tốc độ nạp): Số token được nạp lại mỗi giây.

- **Token Cost** (giá mỗi request): Mỗi request tiêu thụ bao nhiêu token.

---

### ⚙️ Cách Hoạt Động Chi Tiết

**Thiết kế: Token Bucket + Lazy Refill trong Lua Script**

Toàn bộ logic rate limit (refill + check + trừ token) chạy **atomic** trong 1 Lua script trên Redis.
Mỗi request tự tính refill dựa trên thời gian trôi qua kể từ lần cuối → không cần background worker.

Ưu điểm:
- **Atomic:** Lua script chạy single-threaded trên Redis → không race condition
- **Stateless trên Java:** Filter chỉ gọi script, không giữ state
- **Chính xác:** Refill tính theo delta time thực tế, không phụ thuộc scheduler

#### **1. IP-Based Rate Limit (Lua Script)**

Script: `ip_rate_limit.lua` — Gọi bởi `IPBasedRateLimitGlobalFilter` (Order: -100)

```
┌──────────────────────────────┐
│  Request (IP Address)        │
└──────────────┬───────────────┘
               │
       ┌───────▼────────┐
       │  Blacklisted?  │
       └───┬────────┬───┘
          Yes      No
           │        │
    ┌──────▼──┐  ┌──▼──────────────────┐
    │  Block  │  │ Lazy Refill Tokens  │
    └─────────┘  │ (delta × rate)      │
                 └──────────┬──────────┘
                            │
                    ┌───────▼────────┐
                    │ tokens ≥ 1 ?   │
                    └───┬────────┬───┘
                       Yes      No
                        │        │
                 ┌──────▼───┐  ┌──▼──────────────┐
                 │ tokens-1 │  │ Abuse Counter + 1│
                 │  Allow   │  └─────┬───────────┘
                 └──────────┘        │
                              ┌──────▼─────────┐
                              │ counter > 10 ? │
                              └──┬──────────┬──┘
                                Yes        No
                                 │          │
                          ┌──────▼────┐ ┌─────▼──────────┐
                          │ Blacklist │ │   429 Too Many │
                          │   Block   │ │   Requests     │
                          └───────────┘ └────────────────┘
```

#### **2. Account-Based Rate Limit (Lua Script)**

Script: `account_rate_limit.lua` — Gọi bởi `AccountBasedRateLimitGatewayFilterFactory` (Order: 10)

```
┌──────────────────────────────────┐
│  Request (X-User-Id header)      │
└──────────────┬───────────────────┘
               │
    ┌──────────▼───────────────┐
    │  Lazy Refill Tokens      │
    │  (delta_minutes × rate)  │
    └──────────┬───────────────┘
               │
       ┌───────▼────────┐
       │ tokens ≥ 1 ?   │
       └───┬────────┬───┘
          Yes      No
           │        │
    ┌──────▼───┐ ┌──▼──────────────┐
    │ tokens-1 │ │   429 Too Many  │
    │    Allow │ │   Requests      │
    └──────────┘ └─────────────────┘
```

> **So sánh IP vs Account:**
> - IP dùng `refill_rate` đơn vị **tokens/giây**, Account dùng **tokens/phút**
> - IP có thêm **Blacklist** + **Abuse Counter** (chống DDoS), Account thì không
> - Account luôn lưu state kể cả khi reject (để timestamp được cập nhật)

#### **3. Redis Key Strategy**

**IP-Based Keys:**
```
    rate_limit:ip:{IP}          → HASH { tokens, timestamp }
    abuse_counter:ip:{IP}       → HASH { abuse_counter }     TTL: 120s
    blacklist:ip:{IP}           → STRING "1"                  TTL: 120s
```

**Account-Based Keys:**
```
    rate_limit:account:{ID}     → HASH { tokens, timestamp }
```

> **TTL đóng vai trò cleanup:** Khi key hết TTL, Redis tự xoá.
> Lần request tiếp theo sẽ tạo bucket mới với capacity đầy.

## PHASE 4: LOAD BALANCING & RESILIENCE

### 🎯 Mục đích

| **Đặc tính** | **Mô tả** |
|--------------|-----------|
| **High Availability** | Nếu một instance bị sập (crash), hệ thống vẫn hoạt động nhờ các instance còn lại |
| **Scalability** | Khi lượng user tăng vọt, bạn bật thêm instance mới để chịu tải |

---

### ⚙️ Các Thuật Toán Phổ Biến

| **Thuật toán** | **Cơ chế** | **Ưu điểm** | **Nhược điểm / Phù hợp** |
|----------------|------------|-------------|--------------------------|
| **Round Robin**<br>(Vòng tròn) | Gateway phân phối lần lượt theo thứ tự:<br>Request 1 → Instance A<br>Request 2 → Instance B<br>Request 3 → Instance C<br>Request 4 → quay lại Instance A | ✅ Đơn giản, dễ cài đặt | ❌ Không quan tâm đến tình trạng sức khỏe của server<br>(Instance A đang chậm vẫn bị dí request) |
| **Weighted Round Robin**<br>(Vòng tròn có trọng số) | Tương tự Round Robin, nhưng gán "điểm số" cho server.<br>Server mạnh hơn (RAM/CPU cao) nhận trọng số cao hơn | ✅ Tối ưu cho hạ tầng không đồng nhất | 🎯 Phù hợp: Khi có máy mạnh, máy yếu |
| **Least Connections**<br>(Ít kết nối nhất) | Gateway kiểm tra instance nào đang xử lý ít request nhất và đẩy request mới vào đó | ✅ Tối ưu cho request có thời gian xử lý không đều<br>✅ Tránh server bị quá tải | 🎯 Phù hợp: Request có độ phức tạp khác nhau |

---

### 🔍 Kết Hợp Với Service Discovery

```
Backend Service Start → Register IP/Port → Service Registry
                                              ↓
API Gateway Receives Request → Query Registry → Get Active Instances
                                              ↓
                                    Apply Load Balancing Algorithm
                                              ↓
                                    Forward Request to Selected Instance
```

| **Bước** | **Mô tả** |
|----------|-----------|
| **1. Service Discovery** | Khi Backend khởi động, đăng ký IP/Port lên Service Registry (danh bạ chung) |
| **2. API Gateway Query** | Khi nhận request, Gateway hỏi "danh bạ" xem Service A có những instance nào active |
| **3. Load Balancing** | Sau khi có danh sách IP, Gateway dùng thuật toán (VD: Round Robin) để chọn IP và chuyển tiếp |

> 💡 **Lưu ý:** Trong Spring Cloud, Spring Cloud Gateway kết hợp với **Spring Cloud LoadBalancer** để thực hiện tự động.
